# GraphQL Workshop: Fragment-Driven Architecture

> **Goal:** Move from multiple fragmented queries to a "One Query Per Page" architecture using GraphQL Fragments, Apollo Client, and TypeScript codegen.

## Stack

| Layer | Technology |
|---|---|
| Framework | Next.js 14 (App Router) + TypeScript |
| Database | PostgreSQL + Prisma ORM |
| GraphQL Server | Apollo Server 4 + Nexus (code-first) |
| GraphQL Client | Apollo Client 3 |
| Codegen | GraphQL Code Generator (`client-preset`) |

---

## Quick Start

### Option A — Docker (recommended, zero setup)

Requires [Docker Desktop](https://www.docker.com/products/docker-desktop/).

```bash
docker compose up
```

That's it. Docker will:
1. Start a PostgreSQL 16 container
2. Push the Prisma schema
3. Seed 5 people with addresses and friendships (Alice, Bob, Carol, Dave, Eve)
4. Generate the Nexus SDL and Apollo Client types
5. Start the Next.js dev server on port 3000

Open [http://localhost:3000](http://localhost:3000) — you'll see 5 people. Click any one to reach `/person/[id]`.

To stop: `docker compose down`. To wipe the database and start fresh: `docker compose down -v && docker compose up`.

Requires Node 20+ and a running PostgreSQL instance.

## Workshop Steps

### Step 0 — Orientation (10 min)

Before writing any code, open Apollo DevTools (browser extension) and the Network tab.

Load `/person/[id]` and observe:
- One POST to `/api/graphql`
- One response containing `person`, `address`, and `friends`
- DevTools shows the cache structure: `Person:clxxx` with all fields merged

This is the **end state** you are building toward.

### Step 1 — The Broken State (20 min)

> **Goal:** Feel the pain of multiple queries before learning the solution.

Navigate to `/workshop/broken/[id]` with any person ID from the home page. The page is live and demonstrates the anti-pattern in the browser — open the Network tab to see 3 separate GraphQL requests fire. A banner on the page links directly to the gold standard `/person/[id]` for side-by-side comparison.

The source is at `src/app/workshop/broken/[id]/page.tsx`. This file shows the anti-pattern:

```tsx
// ❌ Three separate useQuery calls = three network requests
function BrokenAddressSection({ personId }) {
  const { data, loading } = useQuery(GET_PERSON_ADDRESS, { variables: { id: personId } });
  if (loading) return <p>Loading address…</p>; // visible flicker
  // ...
}

function BrokenFriendsSection({ personId }) {
  const { data, loading } = useQuery(GET_PERSON_FRIENDS, { variables: { id: personId } });
  if (loading) return <p>Loading friends…</p>; // another flicker
  // ...
}
```

**Discussion questions:**
1. Open the Network tab. How many GraphQL POST requests fire when the page loads?
2. What does the user see while the slowest query is in-flight?
3. If you add a `country` field to the Address, how many files do you edit?
4. What happens to the Apollo cache with 3 separate query roots?

### Step 2 — Define Your First Fragment (30 min)

> **Goal:** Move the address fields into a fragment colocated with `AddressDisplay`.

**2a.** Open `src/components/AddressDisplay.tsx`. The fragment is already defined:

```tsx
export const AddressFieldsFragment = graphql(`
  fragment AddressFields on Person {
    address {
      id
      street
      city
      zip
    }
  }
`);
```

**2b.** Open `src/app/person/[id]/page.tsx`. Notice how the page query **spreads** the fragment instead of repeating the fields:

```tsx
export const GET_PERSON_QUERY = graphql(`
  query GetPerson($id: ID!) {
    getPerson(id: $id) {
      id
      name
      email
      ...AddressFields    # ← owned by AddressDisplay
      ...FriendsList      # ← owned by FriendsList
    }
  }
`);
```

**2c.** Pass the opaque fragment data down:

```tsx
// page.tsx
<AddressDisplay person={person} />

// AddressDisplay.tsx — useFragment unmasks it
const data = useFragment(AddressFieldsFragment, person);
```

**Key insight — data masking:** `person` in the page is typed as `FragmentType<typeof AddressFieldsFragment>`. This is intentionally opaque — the page **cannot** accidentally read `person.friends` when passing to `AddressDisplay`. TypeScript will error if it tries.

**Exercise:** Try accessing `person.friends` inside `AddressDisplay` after calling `useFragment`. You should get a TypeScript error: property `friends` does not exist.

export const AddressFieldsFragment = graphql(`

      id

### Step 4 — The Mutation Challenge (30 min)

> This is the most important exercise. It reveals why cache management matters.

After a user adds a friend, the server has new data. We have three options to update the UI:

#### Option A — Full Refetch (Naive)

```tsx
// AddFriendForm.tsx
const [addFriend] = useMutation(ADD_FRIEND_MUTATION, {
  refetchQueries: [{ query: GET_PERSON_QUERY, variables: { id: personId } }],
});
```

Open Network tab and add a friend. You will see a full `GetPerson` query fire — **including re-fetching the address**.

**Ask the group:** Did Alice's home address change when she added a friend? No. So why are we fetching it?

#### Option B — Targeted Refetch (Better)

```tsx
const [addFriend] = useMutation(ADD_FRIEND_MUTATION, {
  refetchQueries: ["GetPersonFriends"], // a query that only fetches friends
});
```

This requires splitting the page back into separate queries — which re-introduces the waterfall problem. This is the trap.

#### Option C — Cache Update (Gold Standard)

The mutation response already contains the updated `friends` list (we request it in the mutation's selection set). Apollo's normalized cache identifies objects by `__typename + id`. When the mutation returns `Person { id, friends }`, Apollo **automatically merges** this into the existing `Person:clxxx` cache entry.

```tsx
// The mutation requests the updated friends list:
const ADD_FRIEND_MUTATION = graphql(`
  mutation AddFriend($personId: ID!, $friendId: ID!) {
    addFriend(personId: $personId, friendId: $friendId) {
      id
      friends {    # ← Apollo merges this into the cache
        id
        name
        email
      }
    }
  }
`);
```

After the mutation completes:
1. Apollo writes `Person:clxxx { friends: [...] }` into the cache
2. Any component subscribed to that cache entry re-renders — including `FriendsList`
3. `AddressDisplay` does **not** re-render because `address` was not touched

Open React DevTools Profiler, add a friend, and check which components re-rendered. You should see only `FriendsList`, not `AddressDisplay`.

**Exercise:** Implement the `cache.modify()` approach for an even more surgical update:

```tsx
const [addFriend] = useMutation(ADD_FRIEND_MUTATION, {
  update(cache, { data }) {
    const newFriend = data?.addFriend;
    if (!newFriend) return;

    cache.modify({
      id: cache.identify({ __typename: "Person", id: personId }),
      fields: {
        friends(existingFriendRefs = []) {
          const newFriendRef = cache.writeFragment({
            data: newFriend,
            fragment: FriendsListFragment,
          });
          return [...existingFriendRefs, newFriendRef];
        },
      },
    });
  },
});
```

This writes **only the friends field** without any network request at all.

## The Mental Model

```
Page query (ONE useQuery)
├── ...AddressFields     ← AddressDisplay.tsx fragment
│     └── address { street city zip }
│
└── ...FriendsList       ← FriendsList.tsx fragment
      └── friends { id name email }
```

**Rule:** A component should never know about fields it doesn't render. If `AddressDisplay` doesn't show `friends`, it shouldn't request them — and with fragment masking, it *can't*.

## Codegen Workflow

Every time the GraphQL schema or a query/fragment in a `.ts`/`.tsx` file changes:

```bash
# If you changed the Nexus schema types:
npm run schema:generate   # updates schema.graphql

# Always run after schema:generate or after editing queries/fragments:
npm run codegen           # updates src/gql/
```

The `src/gql/` directory is gitignored — it's fully reproducible from `schema.graphql` and the query strings in your source files. Treat it like a build artifact.

## Troubleshooting

| Problem | Fix |
|---|---|
| `Cannot find module '@/gql'` | Run `npm run codegen` |
| `Fragment X not found` | Ensure the fragment name in the `graphql()` tag matches exactly |
| `Property 'friends' does not exist` in a component | This is intentional — data masking. Use `useFragment` for the correct fragment. |
| Mutation fires but UI doesn't update | Check that the mutation response requests the same fields as the cached fragment |
| `prisma` is not defined | Run `npm run db:generate` after schema changes |
